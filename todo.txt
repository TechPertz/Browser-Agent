Few Suggestions from gemini on selected topics
If they are evaluating you on *extreme* accuracy, then relying on a generic wrapper or a single zero-shot LLM prompt won't survive contact with reality. Web UIs are chaotic, and standard agents fail because they hallucinate selectors or click before a page finishes loading.

To achieve 95%+ accuracy, you have to treat the browser agent less like a human clicking around, and more like a high-stakes ETL pipeline where the browser is an incredibly volatile data source. Every step must have defensive checks.

Here are the specific, engineering-heavy solutions that push an agent from "demo-quality" to "audit-grade" accuracy:

### 1. Visual & Network Stabilization (The Silent Killer)
Agents constantly fail because they parse the DOM or take a screenshot while a loading spinner is active, or while a React component is still hydrating.
* **The Fix:** Never execute an action based purely on the `domcontentloaded` event. Implement a custom wait function that checks for **Network Idle** (no active network requests for 500ms) *and* **DOM Mutation Idle** (no changes to the HTML structure for 500ms). Ensure the page is visually stabilized before the agent "looks" at it.

### 2. The "Expectation vs. Reality" Checkpoint (Strict State Transitions)
If the agent clicks "Open Pull Request," it shouldn't just blindly proceed to the next extraction step. 
* **The Fix:** Using the LangGraph architecture outlined in your plan, enforce a rigid `VERIFY` node after every single `ACT` node. Before taking an action, the agent must output a predicted state (e.g., `"Expected: A modal containing the CI checks will appear"`). The `VERIFY` node then takes a fresh screenshot and evaluates: *"Did the expected state occur?"* If not, it routes back to a `REFLECT` node to self-correct before the pipeline gets permanently derailed.

### 3. Dual-Grounding (Visual + DOM Cross-Referencing)
Relying solely on Set-of-Mark (bounding boxes) or solely on the DOM accessibility tree leads to misclicks, especially in dense UIs like Jira or GitHub Blame views. 
* **The Fix:** Force the model to cross-reference. Require the LLM to output a JSON payload that includes the bounding box ID *and* the exact text inside that box before clicking. 
    ```json
    {
      "action": "click",
      "target_box_id": "84",
      "verification_text": "build-and-test / py35-build",
      "confidence": 0.98
    }
    ```
    If the agent's `verification_text` doesn't strictly match the text parsed from the underlying DOM element for that box, the action is blocked and re-evaluated.

### 4. Specialized Sub-Agents (Routing)
A single "general" prompt gets confused by complex, nested UI workflows (like Task 3's requirement to dig into specific passed/failed CI checks).
* **The Fix:** Use a "Router" pattern. The Orchestrator LLM detects the context (e.g., "I am on a GitHub Actions page") and hands control over to a specialized sub-agent with a highly specific prompt dedicated *only* to navigating GitHub CI trees. Narrowing the context window drastically reduces hallucination rates.

### 5. Deterministic Fallbacks
LLMs are probabilistic. Data extraction should be as deterministic as possible.
* **The Fix:** If the agent is trying to extract the assignee and due date from a Linear ticket, don't just ask the LLM "What is the date?" Have the LLM locate the DOM node containing the data, and then use standard Python scripts (regex, date parsers) to extract and format the data from that specific node. 

By wrapping the LLM's probabilistic reasoning in rigid, deterministic state-checks, you guarantee that failures are caught and retried, rather than silently passed down the pipeline.
